ix Document Preview and IoT Download (Local Filesystem Fallback)
The document preview on the "Your Order" screen fails to load and remains blank because the Backblaze B2 cloud storage account has exceeded its free download bandwidth / Class B transaction limits, throwing AccessDenied (HTTP 403) errors on file retrieval.

This plan introduces a local filesystem storage fallback in the backend. When files are uploaded or committed, they are saved locally in the uploads/ directory of the server (while still attempting B2 in the background). On retrieval (for previews or downloads), the backend will first check the local directory, bypassing B2 entirely for all newly uploaded files. A local download route will also be provided for the IoT printers.

User Review Required
NOTE

This change is fully backward-compatible. If a file is not found in the local uploads/ directory, the backend will still attempt to fetch it from B2 (in case the file was uploaded in a previous session). If B2 throws a bandwidth cap error, the backend will catch it gracefully instead of crashing.

IMPORTANT

The printer simulator running locally will now fetch files directly from the backend via HTTP rather than the B2 S3 signed URL, which completely resolves the caps exceeded issue for testing prints.

Proposed Changes
Backend Service and Routing
[MODIFY] 
storage.ts
Implement local directory storage in uploads/ inside the backend directory.
Update putObject to write the file locally, then try uploading to B2 in a try-catch block.
Update getObjectBuffer to first check the local filesystem; if not found, fall back to B2.
Update deleteKeys to remove files locally and from B2.
Update listKeys to merge local files matching a prefix and B2 objects.
Update presignGet to return the backend URL http://localhost:4000/api/kiosk/download/:fileKey instead of an S3 presigned URL.
[MODIFY] 
kiosk.ts
Add a public route GET /download/:fileKey to serve the stored files (PDFs, images, docx) to the printer or clients directly from local storage/B2.
